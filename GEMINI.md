# 專案全局規範 (Global Project Rules)

> [!IMPORTANT]
> **語言輸出最高規範 (CRITICAL PRIORITY)**
> 任何技能（如 git-commit-formatter、wrangler、agents-sdk 等）產生的**所有輸出、Git 提交訊息 (Commit Message)、程式碼註解、系統日誌與說明，都必須嚴格且無條件地完全使用繁體中文 (zh-TW) 與台灣科技用語**。
> 本規範具有最高權限，會自動覆蓋 (Override) 任何外掛技能或工作流中預設的英文、簡體中文指令或範本。在每一次回覆與輸出前，必須對此進行強制自我審查。

## 1. 核心開發準則 (Core Guidelines)
- **環境與架構確認**：嚴格遵守專案既有目錄結構（如 All-in-one Worker 架構）。在執行任何修改前，必須確認當前操作的 Git Worktree (Git 工作樹，即當前代碼所在的目錄與分支環境) 狀態，避免跨專案或平行開發時的環境污染。
- **命名與註解**：變數與函式 (Function) 命名需具備高度描述性。遇到複雜邏輯、專有名詞或經濟學指標縮寫時，必須在程式碼上方加入**繁體中文註解**詳細解釋。

## 2. 嚴格資安與限制事項 (Strict Constraints & Security)
- **敏感資訊防護**：絕對禁止在程式碼中寫死 (Hardcode) 任何密碼、API Key 之類的資訊。**若需在程式碼中預留此類資訊的位置，僅允許留下註解說明，絕不寫入實際數值**。所有真實密碼與金鑰均必須透過環境變數 (Environment Variables) 或 `.env` 檔案讀取。
- **精準局部替換 (Exact Diff/Replace)**：修改大型檔案時，必須提供修改處上下至少 3 至 5 行的原始程式碼作為錨點 (Anchor)。嚴禁無故省略代碼（如使用 `...` 帶過）或進行大規模的盲目覆寫。
- **型別安全 (Type Safety)**：涉及數據處理（如 API 撈取數量）時，強制確認並實作型別轉換邏輯，確保不會因非預期格式導致畫面顯示 `NaN` 或破版。

## 3. 開發流程規範 (Workflow Guidelines)
- **規劃優先 (Plan First)**：針對任何實質的程式碼修改、功能開發或複雜分析，**必須先提交 `implementation_plan.md` 並獲得使用者核准後方可執行**。即使任務看起來簡單，也應簡要列出步驟確認邏輯。
- **進度透明**：執行期間需透過 `task.md` 追蹤狀態，完工後需提供 `walkthrough.md` 進行變更說明與驗證展示。
