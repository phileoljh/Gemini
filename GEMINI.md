## 1. 專案編碼規範 (Project Coding Standards)

- **架構遵循**：必須嚴格遵守專案既有的目錄結構與整體架構。
- **命名規範**：變數與函式 (Function) 命名必須具備高度描述性 (Descriptive)，嚴禁使用如 `a`, `b`, `temp`, `data1` 等無意義名稱。
- **註解規範**：遇到特定的經濟學指標縮寫（如 PAYEMS, JTSJOL）、專業術語，或是複雜的處理邏輯（如計算月增率 MoM、年齡均線或 NEW 標籤的基期等），必須在程式碼上方加入**繁體中文註解**進行詳細解釋。

## 2. 嚴格限制與資料安全 (Strict Constraints & Type Safety)

- **敏感資訊防護**：絕對禁止在程式碼中寫死 (Hardcode) 任何 API Key（如 FRED API Key）。所有敏感資訊均必須透過環境變數 (Environment Variables) 或 `.env` 檔案進行讀取。
- **型別安全與修改評估**：
  - 涉及數據處理（如從 API 撈取數量、流量等）時，必須確保資料已正確轉換為數字型別，避免畫面顯示 `NaN` 或導致破版。
  - 在修改現有功能（如增加新指標、調整排版）前，必須先梳理該指標的分類（Leading, Coincident, Lagging）及其顯示格式設定，確保改動不會引發型別轉換錯誤或破壞現有版面。

## 3. 版本控制規範 (Version Control Standards)

- **Git 提交格式**：若需產出 Git 提交紀錄 (Commit Message)，一律強制採用 **Conventional Commits (常規提交)** 格式。
  - **參考範例**：
    - `feat: 新增 VIX 指標`
    - `fix: 修正時間戳記解析錯誤`
    - `docs: 更新開發者 SOP`

## 🚫 絕對禁止事項 (Strict Taboos)

1. **嚴禁「一鍵重構」**：禁止在沒有拆解步驟的情況下，一次性大規模修改多個函數或檔案。
2. **嚴禁「無故省略」與「偷懶」**：禁止使用 `// ... (other code continues)` 或 `// 保持不變` 等省略符號。針對一般檔案，必須提供**完整、可直接替換**的代碼；若遇超大型檔案，必須提供**完整的函數區塊**或使用精準的**局部替換 (Diff/Replace) 格式**，絕不可讓使用者因上下文遺失而覆蓋錯誤。
3. **嚴禁「破壞註解」**：現有的繁體中文註解、文件說明必須 100% 保留。若修改涉及複雜邏輯（如計算月增率 MoM、特定經濟學指標縮寫等），**必須**主動在程式碼上方更新或加入繁體中文註解。
4. **嚴禁 Hardcode 敏感資訊**：所有 API Key 等機密資訊必須透過環境變數 (`process.env`) 讀取，絕對禁止寫死在程式碼中。
