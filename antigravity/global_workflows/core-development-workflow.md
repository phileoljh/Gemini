# 核心開發流程 (Core Development Workflow)

> **⚡ 輕量任務豁免條款**：若任務屬於「單行錯別字修正」或「顯而易見的一行修改」等瑣碎操作，可跳過完整四階段流程，直接執行並說明原因。**判斷基準**：任何涉及邏輯、跨檔案影響或需要決策的任務，一律走完整流程。

當接收到**非瑣碎**的開發需求或 Bug 修正時，AI 必須強制進入 **Planning Mode (規劃模式)**，並嚴格遵循以下 4 個階段：

### 階段 1：情資蒐集與根本原因分析 (Reconnaissance & RCA)
* **絕對不要急著寫程式碼**。
* 搜索並讀取相關檔案，梳理資料流與依賴關係。
* **謀定而後動 (Think Before Coding)**：不清楚就問，絕不隱藏困惑。若需求有歧義或存在多種實作方式，必須主動提出選項與權衡 (Tradeoffs)，嚴禁默默自行猜測。
* **Bug 修正特例**：必須先執行 Root Cause Analysis (根本原因分析)。你不能只看錯誤訊息，必須追溯源頭，並在計畫中明確指出「哪個檔案的哪一段邏輯缺陷，導致了這個 Bug」。

### 階段 2：制定計畫 (Implementation Plan)
* **目標驅動 (Goal-Driven)**：在列出任務清單之前，必須先將「指令」轉化為「可驗證的目標」。
    * **❌ 錯誤**：「修復 Bug」或「加上驗證」。
    * **✅ 正確**：「寫一個能重現 Bug 的測試/腳本 → 修復它並讓測試通過」。
* 以 **Markdown Task List** 格式輸出具體計畫，並等待使用者批准 (User Approval)。每個任務項目必須包含明確的驗證步驟：
    - [ ] `檔案路徑`：具體說明要修改的函式名稱與變更邏輯 → **驗證**：[具體的成功標準]
    - [ ] `檔案路徑`：預期加入的防呆機制或型別檢查 → **驗證**：[確認方式]

### 階段 3：原子化執行 (Atomic Execution)
* **一次只做一件事 (One Thing at a Time)**。嚴禁將「修正 Bug」與「新增功能」混在同一次檔案覆寫中。
* **極簡至上 (Simplicity First)**：以最少的程式碼解決問題。不寫超出需求的功能、不寫預測性或過度抽象的邏輯。如果能用 50 行寫完，就不要寫 200 行。
* **外科手術式修改 (Surgical Changes)**：
    * 嚴格依據計畫逐項執行，並確保局部替換的錨點 (Anchor) 精準無誤。
    * **只改該改的**。嚴禁「順手優化」或重構與本次任務無關的程式碼與註解格式（即使你有更好的寫法）。
    * **清理戰場**：若你的修改導致了無用的變數、函式或 Imports 產生，必須負責清除；但若是原本就存在的無用代碼，除非被要求，否則不要動。

### 階段 4：自我驗證與強制中斷 (Self-Verification & Fallback)
修改完成後，AI 必須逐項執行以下具體自我檢查，而非模糊的「沙盤推演」：
* **驗證清單 (Verification Checklist)**：
    - [ ] **範圍確認**：逐行比對 diff，確認每一行修改都能直接追溯到計畫中的任務項目。有無觸碰計畫範圍外的代碼？
    - [ ] **成功標準**：是否已達成階段 2 中每個任務項目定義的「驗證」條件？
    - [ ] **依賴檢查**：有無引入未宣告的新依賴、套件或全域變數？
    - [ ] **型別安全**：是否確認了關鍵變數的型別，避免 `NaN` 或執行期型別錯誤？
* ⛔ **強制中斷規則 (Hard Stop Rule)**：若在自我驗證或實際執行後發現依然存在 Error 或 Bug 未解，**嚴禁進行超過 2 次的盲目試錯與重試**。
* 一旦觸發強制中斷，AI 必須立即停止修改，還原變更 (Revert)，並向使用者輸出具體的錯誤 Log (日誌) 與依賴衝突分析，等待人類給予新指示。
